# Chapter 2
## 운영체제 서비스
* 운영체제는 프로그램 및 사용자에게 프로그램 및 서비스를 실행할 수 있는 환경을 제공
* 운영체제 서비스 중 하나는 사용자에게 유용한 기능을 제공
	* *사용자 인터페이스*: 거의 모든 운영체제에는 사용자 인터페이스(`UI`)가 있음
		* `Command-Line(CLI)`, `Graphics User Interface(GUI)`, `Batch`
		* Batch -> 한 번에 submit
	* *프로그램 실행*: 시스템이 프로그램을 메모리에 로드하고 해당 프로그램을 실행하거나 정상적으로 또는 비정상적으로 실행을 종료할 수 있어야 함(오류 표시)
	* *IO 작업*: 실행 중인 프로그램에 I/O가 필요할 수 있으며, 여기에는 파일 또는 I/O 장치가 포함될 수 있음
	* *파일 시스템 조작*: 파일 시스템이 특히 중요함. 프로그램은 파일과 디렉터리를 읽고 쓰고, 만들고 삭제하고, 검색하고, 파일 정보를 나열하고, 권한을 관리해야 함
	* *통신*: 프로세스는 네트워크를 통해 동일한 컴퓨터 또는 컴퓨터 간에 정보를 교환할 수 있음
		* 공유 메모리를 통해 또는 메시지 전달을 통해 통신할 수 있음(OS에서 패킷으로 이동)
	* *오류 감지*: OS는 발생할 수 있는 오류를 지속적으로 인식해야 함
		* CPU 및 메모리 하드웨어, I/O 장치, 사용자 프로그램에서 발생할 수 있음
		* 각 오류 유형에 대해 OS는 정확하고 일관된 컴퓨팅을 보장하기 위해 적절한 조치를 취해야 함
		* 디버깅 기능은 사용자와 프로그래머의 시스템을 효율적으로 사용할 수 있는 능력을 크게 향상시킬 수 있음
* 리소스 공유를 통해 시스템 자체의 효율적인 운영을 보장하기 위한 또 다른 OS 기능 집합이 존재함
	* *리소스 할당*: 여러 사용자 또는 여러 작업을 동시에 실행하는 경우 각 사용자에게 리소스를 할당해야 함
		* CPU 주기, 메인 메모리, 파일 스토리지, I/O 장치 등 다양한 유형의 리소스
	* *회계*: 어떤 사용자가 얼마나 많은 컴퓨터 리소스를 사용하고 어떤 종류의 컴퓨터 리소스를 사용하는지 추적
	* *보호 및 보안*: 다중 사용자 또는 네트워크 컴퓨터 시스템에 저장된 정보의 소유자는 해당 정보의 사용을 제어하고자 할 수 있으며, 동시 프로세스가 서로 간섭하지 않아야 함
		* *보호*에는 시스템 리소스에 대한 모든 액세스가 제어되도록 하는 작업이 포함됨
		* 외부로부터의 시스템 *보안*은 사용자 인증이 필요하며, 잘못된 액세스 시도로부터 외부 I/O 장치를 보호하는 데까지 확장

## 사용자 운영체제 인터페이스 - CLI
* CLI 또는 *명령 인터프리터*가 직접 명령 입력을 허용함
	* 커널에 구현되기도 하고, 시스템 프로그램에 의해 구현되기도 함
	* 때로는 여러 가지 맛이 구현되기도 함 - *shells*
	* 주로 사용자로부터 명령을 가져와 실행함
	* 때로는 기본 제공 명령, 때로는 프로그램 이름만
		* 후자의 경우 새 기능을 추가하는데 shell 수정이 필요하지 않음

## 사용자 운영체제 인터페이스 - GUI
* 사용자 친화적인 *데스크탑* 메타포 인터페이스
	* 일반적으로 마우스, 키보드 및 모니터
	* *아이콘*은 파일, 프로그램, 작업 등을 나타냄
	* 인터페이스의 개체 위에 있는 다양한 마우스 버튼으로 인해 다양한 작업(정보 제공, 옵션, 실행 기능, 열린 디렉토리(폴더라고 함)이 발생)
	* Xerox PARC에서 발명
* 현재 많은 시스템에 CLI 및 GUI 인터페이스 모두 포함
	* Microsoft Windows는 CLI “command” shell이 포함된 GUI
	* 애플 맥 OS X는 아래에 유닉스 터널을 포함하고 shell을 사용할 수 있는 “Aqua” GUI 인터페이스
	* Unix 및 Linux에는 선택적 GUI 인터페이스(CDE, KDE, GNOME)가 포함된 CLI가 있음

## 시스템 콜
* OS에서 제공하는 서비스에 대한 프로그래밍 인터페이스
* 일반적으로 고급 언어(C, C++)로 작성
* 직접적인 시스템 호출이 아닌 고급 *API(Application Programming Interface)*를 통해 프로그램이 대부분 액세스
* 가장 일반적인 세 가지 API는 윈도우용 Win32 API, POSIX(Portable OS Interface) 기반 시스템용 POSIX API, 자바 가상 머신(JVM)용 Java API

## 시스템 콜 구현
* 일반적으로 각 시스템 콜과 관련된 번호
	* *시스템 콜 인터페이스*는 이 숫자에 따라 색인화된 테이블을 유지
* 시스템 콜 인터페이스는 OS 커널에서 의도한 시스템 콜을 호출하고 시스템 콜의 상태와 모든 반환 값을 반환
* 호출자는 시스템 콜이 어떻게 구현되는지 알 필요가 없음
	* API는 준수하고 OS가 결과 호출로 어떤 작업을 수행할지 파악하기만 하면 됨
	* API에 의해 프로그래머에게 숨겨진 대부분의 OS 인터페이스 세부 정보
		* 런터임 지원 라이브러리에서 관리(컴파일러에 포함된 라이브러리에 내장된 기능 집합)
## 시스템 콜 매개 변수 전달
* 종종 원하는 시스템 호출의 단순한 식별 정보보다 더 많은 정보가 필요
	* 정확한 정보 유형과 양은 OS 및 호출에 따라 다름
* OS에 매개 변수를 전달하는 데 사용되는 세 가지 일반적인 방법
	* 단순: 레지스터의 매개 변수를 전달
		* 경우에 따라 레지스터보다 많은 매개 변수가 있을 수 있음
	* 메모리에 있는 블록 또는 테이블에 저장된 매개 변수 및 레지스터에서 매개 변수로 전달된 블록 주소
		* Linux 및 Solaris가 채택한 접근 방식
	* 프로그램에 의해 *스택*에 배치되거나 *푸시*되고 운영체제에 의해 스택에서 *분리*되는 매개 변수
	* 블록 및 스택 메서드는 전달되는 매개 변수의 수 또는 길이를 제한하지 않음

## 시스템 콜 타입
* 프로세스 컨트롤
	* 프로세스 생성, 프로세스 종료(fork, exit)
	* 종료, 중단
	* 로드, 실행
	* 프로세스 특성 가져오기, 프로세스 특성 설정
	* 시간을 기다림
	* 대기 이벤트, 신호 이벤트
	* 메모리 할당 및 사용 가능
	* 오류인 경우 메모리 덤프
	* *버그*를 확인하는 *디버거*, *단일 단계* 실행
	* 프로세스 간 공유 데이터에 대한 액세스를 관리하기 위한 *잠금*
* 파일 관리
	* 파일 생성, 파일 삭제
	* 파일 열기, 닫기
	* 읽기, 쓰기, 재배치
	* 파일 특성 가져오기 및 설정
* 장치 관리(I/O device)
	* 요청 장치, 해제 장치
	* 읽기, 쓰기, 재배치
	* 장치 특성 가져오기, 장치 특성 설정
	* 디바이스를 논리적으로 연결 또는 분리
* 정보 유지 관리
	* 시간 또는 날짜 가져오기, 설정
	* 시스템 데이터 가져오기, 설정
	* 프로세스, 파일 또는 장치 특성 가져오기, 설정
* 커뮤니케이션
	* 통신 연결 생성, 삭제
	* *호스트 이름* 또는 *프로세스 이름*으로 *모델 전달 메시지* 전송, 메시지 수신
		* *클라이언트*에서 *서버*로
	* *공유 메모리 모델*은 메모리 영역을 생성하고 액세스
	* 상태 정보를 전송
	* 원격 디바이스 연결 및 분리	
* 보호
	* 리소스에 대한 액세스 제어
	* 사용 권한 가져오기 및 설정
	* 사용자 액세스 허용 및 거부

## MS-DOS
* 가장 초기 OS
* Single-tasking -> 한 메모리 안에 하나의 프로세스만 둘 수 있음
* 시스템 부팅 시 shell 호출
* 간단한 프로그램 실행 방법
	* 프로세스가 생성되지 않음
* 단일 메모리 공간
* 커널을 제외한 모든 프로그램을 덮어쓰면서 프로그램을 메모리에 로드
* 프로그램 종료 -> shell 다시 로드

## FreeBSD
* 유닉스 계열
* 멀티테스킹 -> 한 메모리 안에 여러개의 프로세스가 동시에 생성될 수 있음
* 사용자 로그인 -> 사용자가 선택한 shell을 호출
* shell이 `fork()` 시스템 콜을 실행하여 프로세스 생성
	* `exec()`을 실행하여 프로그램을 프로세스에 로드
	* shell은 프로세스가 종료될 때까지 대기하거나 사용자 명령을 계속함
* 프로세스가 종료되는 시간
	* code = 0 - no error
	* code > 0 - error code

## 시스템 프로그램
* 시스템 프로그램은 프로그램 개발과 실행을 위한 편리한 환경을 제공
	* 파일 처리
	* 상태 정보가 파일 수정에 저장되는 경우가 있음
	* 프로그래밍 언어 지원
	* 프로그램 로드 및 실행
	* 커뮤니케이션
	* 백그라운드 서비스
	* 응용 프로그램
* 운영체제에 대한 대부분의 사용자의 견해의 실제 시스템 콜이 아닌 시스템 프로그램에 의해 정의
* 프로그램 개발 및 실행을 위한 편리한 환경 제공
	* 이들 중 일부는 단순히 시스템 콜에 대한 사용자 인터페이스이며, 다른 일부는 상당히 복잡함

## 실행(Implementation)
* 많은 변화
	* 어셈블리어로 된 초기 OS
	* Algol, PL/1과 같은 시스템 프로그래밍 언어
	* 지금은 C, C++
* 보통 여러 언어를 섞음
	* 어셈블리 중 가장 낮은 수준
	* 메인은 C
	* C, C++의 시스템 프로그램, PERL, Python, shell 스크립트 등의 스크립트 언어
* 다른 하드웨어로 쉽게 이식할 수 있는 고급 언어
	* 그러나 느림
* Emulation을 통해 OS를 non-native 하드웨어에서 실행할 수 있음

## 운영체제 구조
* 범용 OS는 매우 큰 프로그램
* 다양한 구조화 방법
	* 간단한 구조: MS-DOS
	* 더 복잡한: UNIX
	* 계층화: 추상화
	* Microkernel: Mach

## 간단한 구조 - MS-DOS
* MS-DOS: 최소 공간에서 최대한의 기능을 제공하도록 작성됨
	* 모듈로 분할되지 않음
	* MS-DOS는 일부 구조를 가지고 있지만, 그것의 인터페이스와 기능 수준은 잘 분리되어 있지 않음

## 간단하지 않은 구조 - UNIX
* UNIX: 하드웨어 기능에 의해 제한되는, 원래의 유닉스 운영체제의 구조가 제한. UNIX OS는 두 개의 분리 가능한 부분으로 구성
	* 시스템 프로그램
	* The kernel
		* 시스템 콜 인터페이스 아래 및 물리적 하드웨어 위에 있는 모든 구성
		* 파일 시스템, CPU 스케줄링, 메모리 관리 및 기타 운영체제 기능 제공, 한 레벨에 대한 많은 기능 제공
