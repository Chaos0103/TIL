# Chapter 3: 프로세스
#운영체제

## 프로세스 개념
* 운영체제는 다양한 프로그램을 실행
	* 배치 시스템 - *작업*
	* 시간 공유 시스템 - *사용자 프로그램* 또는 *작업*
* *프로세스*: 실행 중인 프로그램, 프로세스 실행은 순차적으로 진행되어야 함
* Multiple parts
	* *텍스트 섹션*이라고도 하는 프로그램 코드
	* *프로그램 카운터*, 프로세서 레지스터를 포함한 현재 활동
	* 임시 데이터가 들어 있는 *스택*
		* 함수 매개변수, 반환 주소, 로컬 변수
	* 전역 변수가 들어 있는 *데이터 섹션*
	* 실행 시간 동안 동적으로 할당된 메모리를 포함하는 *힙*
* 프로그램이 디스크(*실행 파일*)에 저장된 passive 엔티티이며 프로세스가 활성 상태
	* 실행 파일을 메모리에 로드할 때 프로그램이 프로세스로 전환됨
* GUI 마우스 클릭, 해당 이름의 명령줄 입력 등을 통해 프로그램 실행
* 하나의 프로그램은 여러 과정일 수 있음
	* 동일한 프로그램을 실행하는 여러 사용자 고려

## 프로세스 상태
* 프로세스가 실행되면 *상태*가 바뀜
	* *new*: 프로세스가 생성되고 있음
	* *running*: 명령이 실행 중
	* *waiting*: 프로세스가 일부 이벤트가 발생하기를 기다리고 있음
	* *ready*: 프로세스가 프로세서에 할당되기를 기다리고 있음
	* *terminated*: 프로세서가 실행을 마침

## Process Control Block (PCB)
* 각 프로세서와 관련된 정보(작업 제어 블록이라고도 함)
* 프로세서 상태: 실행 중, 대기 중 등
* 프로그램 카운터: 다음에 실행할 명령 위치
* CPU 레지스터: 모든 프로세스 중심 레지스터 내용
* CPU 스케줄링 정보: 우선 순위, 대기열 포인터 스케줄링
* 메모리 관리 정보: 프로세스에 할당된 메모리
* 회계 정보: 사용된 CPU, 시작 후 경과된 클럭 시간, 제한 시간
* IO 상태 정보: 프로세스에 할당된 IO 장치, 열려 있는 파일 목록

## Threads
* 지금까지는 단일 실행 쓰레드인 프로세스만 고려
* 프로세스당 프로그램 카운터가 여러 개인 것을 고려
	* 여러 위치를 동시에 실행할 수 있음
		* 다중 제어 쓰레드 -> threads
* 그런 다음 쓰레드 세부 정보를 저장할 수 있어야 함
* PCB에 여러 프로그램 카운터가 있어야 함

## 프로세스 스케줄링
* CPU 사용 최대화, 시간 공유를 위해 프로세스를 CPU로 신속하게 전환
* *프로세스 스케줄러*는 CPU에서 다음 실행을 위해 사용 가능한 프로세스 중 하나를 선택
* 프로세스 예약 대기열 유지
	* *Job queue*: 시스템의 모든 프로세스 집합
	* *Ready queue*: 기본 메모리에 상주하며 실행 대기 중인 모든 프로세스 집합
	* *Device queues*: I/O 장치를 기다리는 프로세스 집합
	* 프로세스가 다양한 대기열 간에 마이그레이션됨

## 스케줄러
* *Short-term scheduler*(또는 *CPU 스케줄러*): 다음에 실행할 프로세스를 선택하고 할당
	* 때때로 시스템에서 유리한 스케줄러
	* 단기 스케줄러가 자주 호출(ms) -> (빠른 시간이여야 함)
* *Long-term scheduler*(또는 *job 스케줄러*): ready queue에 넣을 프로세서를 선택
	* 장기 스케줄러가 자주 호출되지 않음(초, 분) -> (느릴 수 있음)
	* 장기 스케줄러는 *다중 프로그래밍의 정도*를 제어
* 프로세스는 다음 중 하나로 설명
	* *IO 바인딩된 프로세스*: 계산보다 IO를 수행하는 데 더 많은 시간이 소요되며 CPU가 짧은 경우가 많음
	* *CPU 바인딩된 프로세스*: 계산에 더 많은 시간을 소비하며 매우 긴 CPU 버스트는 거의 없음
* 장기 스케줄러는 좋은 프로세스 조합을 위해 노력

## Medium Term Scheduling
* 다중 프로그래밍의 정도를 줄여야 할 경우 *중간 기간 스케줄러*를 추가할 수 있음
	* 메모리에서 프로세스 제거, 디스크에 저장, 디스크에서 다시 가져와 실행을 계속함: *swapping*

## Context Switch
* CPU가 다른 프로세스로 전환될 때 시스템은 이전 프로세스의 상태를 저장하고 컨텍스트 스위치를 통해 새 프로세스의 저장된 상태를 로드
* PCB에 표시되는 프로세스의 컨텍스트
* 컨텍스트 전환 시간이 오버헤드: 전환 중에는 시스템이 유용한 작업을 수행하지 않음
	* OS와 PCB가 복잡할수록 컨텍스트 스위치가 길어짐
* 하드웨어 지원에 따라 시간이 달라짐
	* 일부 하드웨어는 CPU 당 여러 레지스터 집합을 제공 -> 한 번에 로드되는 다중 컨텍스트

## 프로세스 생성
* 상위 프로세스는 하위 프로세스를 생성하고, 하위 프로세스는 다른 프로세스를 생성하여 프로세스 트리를 형성
* 일반적으로 프로세스 식별자(pid)를 통해 식별 및 관리되는 프로세스
* 리소스 공유 옵션
	* 상위 및 하위 리소스 공유
	* 자녀는 부모 리소스의 하위 집합을 공유
	* 상위 및 하위 공유 리소스 없음
* 실행 옵션
	* 상위 및 하위 항목이 동시에 실행됨
	* 부모는 자식들이 종료될 때까지 기다림
* 주소 공간
	* 부모의 하위 중복
	* 자식이 프로그램을 로드
* UNIX 예제
	* `fock()` 시스템 호출을 통해 새로운 프로세스 생성
	* 프로세스의 메모리 공간을 새 프로그램으로 바꾸기 위해 `fork()` 뒤에 사용되는 `exec()`시스템 호출

## 프로세스 종료
* 프로세스는 마지막 문을 실행한 다음 `exit()` 시스템 콜을 사용하여 운영체제에 삭제 요청
	* 하위에서 상위(`wait()`를 통해) 상태 데이터를 반환
	* 운영체제에 의해 프로세스 리소스가 할당 해제
* 부모는 `abort()` 시스템 콜을 사용하여 자식 프로세스의 실행을 종료할 수 있음
	* 하위 리소스가 할당된 리소스를 초과
	* 하위 항목에 할당된 작업이 더 이상 필요하지 않음
	* 상위 항목이 종료되고 있으며 운영체제에서 상위 항목이 종료되는 경우 하위 항목이 계속되도록 허용하지 않음
* 일부 운영체제에서는 부모가 종료된 경우 하위 항목이 존재하도록 허용하지 않음. 프로세스가 종료되면 프로세스의 모든 자식도 종료해야 함
	* 계단식 종료: 모든 자식, 손자 등은 종료됨
	* 종료는 운영체제에 의해 시작
* 상위 프로세스는 `wait()` 시스템 콜을 사용하여 하위 프로세스의 종료를 기다릴 수 있음. 호출이 종료된 프로세스의 상태 정보와 PID를 반환
	* `pid = wait(&status);`
* *zombie*: 부모가 `wait()`를 호출하지 않은 경우
* *orphan*: 부모가 `wait()`를 호출하지 않고 종료된 경우

## 프로세스 간 통신
* 시스템 내의 프로세스는 /독립적/이거나 /협력적/일 수 있음
* 협력 프로세스는 데이터 공유를 비롯한 다른 프로세스에 영향을 미치거나 영향을 받을 수 있음
* 프로세스가 협력하는 이유
	* 정보 공유
	* 계산 속도 향상
	* 모듈성
	* 편리
* 협력 프로세스에는 *프로세스 간 통신(IPC)* 필요
* IPC의 두 가지 모델
	* *Shared memory*
	* *Message passing*

## 프로세스 협력
* /독립적/ 프로세스는 다른 프로세스의 실행에 영향을 받거나 영향을 받을 수 없음
* /협력/ 프로세스는 다른 프로세스의 실행에 영향을 받거나 영향을 받을 수 있음
* 프로세스 협력의 이점
	* 정보 공유
	* 계산 속도 향상
	* 모듈성
	* 편리

## Producer-Consumer 문제
* 프로세서 협력을 위한 패러다임, 생산자 프로세스는 소비자 프로세스에 의해 소비되는 정보를 생산
	* *unbounded-buffer*는 버퍼의 크기에 실질적인 제한을 두지 않음
	* *제한된 버퍼*의 크기가 있다고 가정

## 프로세스 간 통신 - 공유 메모리
* 통신하고자 하는 프로세스 간에 공유되는 메모리 영역
* 통신은 운영체제가 아닌 사용자 프로세스에 의해 제어
* 주요 문제는 사용자 프로세스가 공유 메모리에 액세스할 때 동작을 동기화할 수 있는 메커니즘을 제공하는 것

## 프로세스 간 통신 - 메시지 패싱
* 프로세스를 통신하고 작업을 동기화하는 메커니즘
* 메시지 시스템: 공유 변수에 의존하지 않고 서로 통신하는 프로세스
* IPC 기능은 다음과 같은 두 가지 작업을 제공
	* `send(message)`
	* `receive(message)`
* 메기지 크기가 고정 또는 가변

## 동기화
* 메시지 전달은 blocking 또는 non-blocking일 수 있음
* *blocking*은 *동기식*으로 간주
	* blocking send: 메시지가 수신될 때까지 발송인이 차단됨
	* blocking receive: 메시지를 사용할 수 있을 때까지 수신기가 차단됨
* *non-blocking*은 *비동기식*으로 간주
	* non-blocking send: 발신인이 메시지를 발송한 후 계속함
	* non-blocking receive: 수신기는 다음을 수신
		* 유효한 메시지
		* Null 메시지
* 다른 조합 가능
	* 송신과 수신이 모두 차단되면, 우리는 *rendezvous*를 함
