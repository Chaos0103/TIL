## 기본 개념
* 다중 프로그래밍을 통해 얻을 수 있는 최대 CPU 활용률
* CPU-IO Burst Cycle: 프로세스 실행은 CPU 실행 및 I/O 대기 *사이클*로 구성
* *CPU Burst*에 이어 *I/O Burst*가 발생
* CPU Burst 분포가 주요 관심사

## CPU 스케줄러
* *단기 스케줄러*는 ready queue에 있는 프로세스 중 하나를 선택하고 CPU를 그 중 하나에 할당
	* 큐는 다양한 방법으로 정렬할 수 있음
* CPU 스케줄링 결정은 다은과 같은 프로세스에서 수행될 수 있음
	1. 실행 상태에서 대기 상태로 전환
	2. 실행 상태에서 준비 상태로 전환
	3. 대기 상태에서 준비 상태로 전환
	4. 종료
* 1과 4에 따른 스케줄링은 *nonpreemptive*
* 다른 모든 스케줄링은 *preemptive*
	* 공유 데이터에 대한 액세스 고려
	* kernel 모드에 있는 동안 preemption 고려
	* 중요한 OS 활동 중에 발생하는 interrupt 고려

## Dispatcher
* dispatcher 모듈은 단기 스케줄러에 의해 선택된 프로세스에 CPU를 제어
	* switching context
	* 사용자 모드로 전환
	* 사용자 프로그램의 적절한 위치로 점프하여 해당 프로그램 다시 시작
* *디스패치 지연 시간*: dispatcher가 한 프로세스를 중지하고 다른 실행을 시작하는데 걸리는 시간

## Scheduling Criteria
* CPU 활용률: CPU 사용률 유지
* 처리량: 시간 단위당 실행을 완료한 프로세스 수
* 처리 시간: 특정 프로세스를 실행하는 데 걸리는 시간
* 대기 시간: 프로세스가 준비 대기열에 대기한 시간
* 응답 시간: 요청이 제출된 시점부터 첫 번째 응답이 생성될 때가지 걸리는 시간(시간 공유 환경의 경우)

## Shortest-Job-First(SJF) 스케줄링
* 각 프로세스에 다음 CPU 버스트의 길이 연결
	* 이 길이를 사용하여 가장 짧은 시간으로 스케줄링
* SJF가 최적: 주어진 프로세스 집합에 대한 최소 평균 대기 시간 제공
	* 문제는 다음 CPU 요청의 길이를 아는 것
	* 사용자에게 물어볼 수 있음

## 우선순위 스케줄링
* 우선 순위 번호(정수)가 각 프로세스와 연결
* CPU는 가장 높은 우선 순위(가장 작은 정수 -> 가장 높은 우선 순위)를 가진 프로세스에 할당
	* Preemptive
	* Nonpreemptive
* SJF는 우선 순위 스케줄링이며 여기서 우선 순위는 예측된 다음 CPU 버스트 시간의 역순
* 문제 = *Starvation*: 우선순위가 낮은 프로세스가 절대 실행되지 않을 수 있음
* 솔루션 = *Aging*: 시간이 지남에 따라 프로세스의 우선 순위가 높아짐

## Round Robin(RR)
* 각 프로세스에는 CPU 시간(*time quantum* q)의 작은 단위(일반적으로 10-100ms)가 할당. 이 시간이 경과하면 프로세스가 우선 처리되고 준비 대기열 끝에 추가
* 준비 대기열에 n개의 프로세스가 있고 time quantum이 q인 경우, 각 프로세스는 최대 q시간 단위의 청크로 CPU 시간의 1/n을 동시에 얻음. (n-1)/q 시간 단위 이상 대기하는 공정이 없음
* 타이머가 다음 프로세스를 예약하기 위해 모든 quantum을 중단
* 성능
	* q large -> FIFO
	* q small -> 컨텍스트 스위치와 관련하여 q가 커야함. 그렇지 않으면 오버헤드가 너무 높음

## Multilevel Queue
* ready queue는 다음과 같이 별도의 대기열로 분할
	* *foreground* (interactive)
	* *background* (batch)
* 지정된 큐에서 영구적으로 처리
* 각 큐에는 고유한 스케줄링 알고리즘이 있음
	* foreground: RR
	* background: FCFS
* 큐 간에 스케줄링을 수행해야 함
	* 고정 우선 순위 스케줄링: (즉, 모든 서비스를 백그라운드에서 제공) starvation의 가능성
	* Time slice: 각 큐는 프로세스 간에 일정할 수 있는 특정 CPU 시간을 얻음(즉, 80%는 RR에 foreground됨
	* FCFS에서 20% 백그라운드 지원

## Multilevel Feedback Queue
* 프로세스는 다양한 큐 사이를 이동할 수 있으며, aging은 이러한 방식으로 구현될 수 있음
* 다음 매개 변수로 정의된 Multilevel Feedback Queue 스케줄러
	* 큐의 수
	* 각 큐에 대한 알고리즘 스케줄링
	* 프로세스 업그레이드 시기를 결정하는 데 사용되는 방법
	* 프로세스 강등 시기를 결정하는 데 사용되는 방법
	* 프로세스가 서비스를 필요로 할 때 어떤 큐에 진입할지 결정하는데 사용되는 방법

## Multiple-Processor 스케줄링
* CPU를 여러 개 사용할 수 있는 경우 CPU 스케줄링의 복잡성 증가
* 멀티프로세서 내의 *동종 프로세서*
* *Asymmetric 멀티프로세싱*: 하나의 프로세서만 시스템 데이터 구조에 액세스하여 데이터 공유 필요성 완화
* *Symmetric 멀티프로세싱(SMP)*: 각 프로세서가 자체 스케줄링되며, 모든 프로세스가 공통 ready queue에 있거나, 각 프로세서가 준비 프로세스의 전용 큐를 가짐
* *프로세서 affinity*: 프로세스가 현재 실행 중인 프로세서에 대한 선호도를 가짐
	* *soft affinity*
	* *hard affinity*
	* *프로세서 세트*를 포함한 변형

## Multiple-Processor 스케줄링 - Load Balancing
* SMP의 경우, 효율성을 위해 모든 CPU를 로드한 상태로 유지해야 함
* 워크로드를 균등하게 분산하기 위한 *load balancing* 시도
* *push migration*: 주기적인 작업이 각 프로세서의 로드를 확인하고, 작업이 오버로드된 CPU에서 다른 CPU로 푸시
* *Pull migration*: idle 프로세서가 사용 중인 프로세서에서 대기 중인 작업 풀
