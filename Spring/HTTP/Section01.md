# 인터넷 네트워크

## 인터넷 통신

- 통신을 주고 받을 때 복잡한 인터넷 망을 거쳐 전달이 됨

## IP(Internet Protocol)

### 인터넷 프로토콜의 역할

- 지정한 IP 주소(IP Address)에 데이터 전달
- 패킷(Packet)이라는 통신 단위로 데이터 전달

### IP 패킷 정보
  
- IP 패킷은 출발지 IP, 목적지 IP, 기타 정보로 구성되어 있으며, 전송 데이터를 감싸고 있음

### IP 프로토콜의 한계

- **비연결성**
  - 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송
- **비신뢰성**
  - 중간에 패킷이 사라지는 경우
  - 패킷이 순서대로 도착을 안하는 경우
- **프로그램 구분**
  - 같은 IP를 사용하는 서버에서 통신하는 애플리케이션이 둘 이상인 경우

## TCP, UDP

### 인터넷 프로토콜 스택의 4계층

- 애플리케이션 계층: HTTP, FTP
- 전송 계층: TCP, UDP
- 인터넷 계층: IP
- 네트워크 인터페이스 계층

### TCP 패킷 정보

- TCP 패킷은 출발지 PORT, 목적지 PORT, 전송 제어, 순서, 검증 정보 등으로 구성되어 있음
- 전송 데이터를 TCP 패킷으로 감싸고, 그 위를 IP 패킷으로 감싸고 있음

### 전송 제어 프로토콜(TCP: Transmission Control Protocol) 특징

- 연결 지향 - TCP 3 way handshake(가상 연결)
- 데이터 전달 보증
- 순서 보장
- 신뢰할 수 있는 프로토콜
- 현재는 대부분 TCP 사용

#### TCP 3 way handshake

1. 클라이언트는 서버에게 SYN을 보내 연결을 요청
2. 서버는 클라이언트에게 SYN와 ACK를 보내 응답
3. 클라이언트는 서버에게 ACK를 보내 응답
4. 데이터 전송
- SYN: 접속 요청
- ACK: 요청 수락
- 참고: 클라이언트가 서버로 보내는 ACK와 함께 데이터 전송 가능

#### 데이터 전달 보증

1. 클라이언트는 서버에게 데이터 전송
2. 서버는 클라이언트에게 데이터를 잘 받았다고 응답

#### 순서 보장

1. 클라이언트는 서버에게 패킷1, 패킷2, 패킷3 순서로 전송
2. 서버에 패킷1, 패킷3, 패킷2 순서로 도착
3. 서버는 클라이언트에게 패킷2부터 다시 보내라고 요청

### UDP 특징

- 사용자 데이터그램 프로토콜(User Datagram Protocol)
- 하얀 도화지에 비유(기능은 거의 없음)
- 연결지향 - TCP 3 way handshake X
- 데이터 전달 보증 X
- 순서 보장 X
- 데이터 전달 및 순서가 보장되지 않지만, 단순하고 빠름
- 정리
  - IP와 거의 같음 + PORT, checksum 정도만 추가
  - 애플리케이션에서 추가 작업 필요

## PORT

- 같은 IP내에서 프로세스 구분
- 0 ~ 65535: 할당 가능
- 0 ~ 1023: 잘 알려진 포트, 사용하지 않는 것이 좋음
  - FTP: 20, 21
  - TELNET: 23
  - HTTP: 80
  - HTTPS: 443

## DNS

### IP의 단점

- 기억하기 어려움
- 같은 컴퓨터여도 사용할 때 마다 IP가 바뀜

### 도메인 네임 시스템(DNS: Domain Name System)

- 도메인 명을 IP 주소로 변환

### DNS 사용

1. 클라이언트는 도메인 명 google.com으로 DNS 서버에 요청하여 IP를 추적
2. DNS서버는 google.com에 맞는 IP 200.200.200.2를 클라이언트에 전달
3. 클라이언트는 DNS 서버로부터 전달받은 200.200.200.2로 서버에 접속
